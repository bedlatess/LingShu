# 灵枢 v1.5.2 稳定性与体验改进方案

## 0. 本次目标

本方案只针对线上使用中暴露出的稳定性、提示体验、管理效率和协议识别问题做收口优化，不扩展臃肿功能。

必须继续遵守既有铁律：

1. 计费语义不变：`charge = base_cost × rate_multiplier`，钱相关路径不重写、不绕开。
2. 用户端不暴露 `base_cost`、`rate_multiplier`、`gross_profit`，只展示用户应付价格、余额、消费和调用记录。
3. 不新增在线支付、分销、多级权限、增长系统、2FA、分组、图片/TTS 等已明确推后的范围。
4. 保留 v1.3 流式真实 usage 回灌、v1.4 上游错误透传、v1.5 协议严格探针。

## 1. 现状判断

根据当前代码初查，用户反馈的问题基本能对应到以下根因：

- 提示重复：管理端大量分散调用 `message.success/error/warning`，父组件 `runWrite` 和子页面也会各弹一次；用户端也存在 `toast` 分散调用。React `StrictMode` 在开发环境会放大 effect 重复执行，但线上重复更多来自缺少统一通知闸门和按钮 loading 防连点。
- 老 API Key 删除报错：用户端删除新 key 正常，老 key 报错，说明不是基础接口全坏，更可能是历史数据、硬删除/软删除不一致、Redis 缓存失效不完整、后端错误映射不统一或前端删除后列表未及时重拉。
- 渠道最近成功时间过长：管理端列表和详情页直接渲染 `last_success_at` 原始值，没有统一时间格式化。
- Anthropic 探测容易误解：后端 Anthropic 实际请求需要 `/v1/models`，但用户填写的标准 `base_url` 通常不带 `/v1`。当前检测接口只传 `base_url + api_key`，没有传“当前选择的协议”，自定义域名时会先尝试 OpenAI 探测，可能把地址归一化成 `/v1` 并识别为 OpenAI。
- 管理端刷新闪登录页：管理端只有 `token` 和 `me` 两个状态。刷新时 token 已存在、me 还没拉回来，于是先显示登录页，稍后 `me` 回来又进入后台。
- 数据不实时：很多列表、详情、弹窗修改后只刷新局部或只刷新父级，缺少统一的“写操作后失效并重拉”规则。
- 额外风险：管理端存在中文乱码；共享类型和用户门户服务里仍能看到 `rate_multiplier` 字段，虽然 DTO 可能剥离，但用户侧后端结构应彻底去掉倍率字段，降低泄漏风险。

## 2. P0 必修：不修会继续影响线上使用

### P0-1 统一提示系统，消除重复 Toast / Message

目标：同一个用户动作只出现一次成功或失败提示。

改法：

1. 管理端新增统一通知封装，例如 `notifySuccess`、`notifyError`、`runWriteOnce`。
2. `runWrite` 只负责失败提示；成功提示由发起动作处负责，或反过来统一由 `runWriteOnce` 接收 `successText`，二选一，禁止父子同时弹。
3. Ant Design 全局配置 `message.config({ maxCount: 1 })`，同类短时间错误使用稳定 key 更新，不堆叠。
4. 所有写按钮增加 loading/disabled 防连点，尤其删除、停用、检测、创建、导入、清理。
5. 用户端 Sonner 使用固定 toast id，例如 `toast.success("...", { id: "api-key-delete" })`，并避免登录页和 AuthProvider 同时提示同一个错误。

验收：

- 登录失败、删除密钥、创建渠道、检测渠道、保存模型、清理日志，每个动作只弹一次提示。
- 连续双击删除/创建按钮不会发出两次请求，不会出现两条成功提示。

### P0-2 修复用户端老 API Key 删除报错

目标：用户能稳定删除自己名下的新旧 API Key，删除后列表立即更新。

改法：

1. 后端统一 API Key 删除语义：用户端删除应走软删除，条件包含 `id + user_id + deleted_at IS NULL`，和管理端保持一致。
2. 对历史已删除或不存在的 key，后端返回稳定中文错误；若产品选择幂等删除，也可以返回 200，但必须保证列表里消失。
3. 删除、停用、重命名前先按 key id 查询 key hash，删除后清理 Redis `apikey:{hash}` 缓存。当前 `InvalidateByID` 在 key 已不存在或硬删除后可能找不到 hash，应先取 hash 再更新/删除。
4. 前端删除成功后先乐观移除当前项，再后台重拉列表；失败时展示后端真实错误，不能只显示泛化报错。
5. 补测试覆盖：新 key 删除、历史软删除 key 再删、非本人 key 删除、删除后网关认证缓存失效。

验收：

- 老 key、新 key 都能删除或得到明确“已删除/不存在”的中文提示。
- 删除后无需刷新页面，列表立即消失。
- 被删除 key 不能继续调用网关。

### P0-3 管理端刷新不再闪登录页

目标：刷新页面、服务器重启后，用户看到“正在校验登录状态”，不闪回登录页。

当前原因：

- 页面刷新时 `token` 从 localStorage 读到了，但 `me` 初始为 `null`。
- 当前渲染条件是 `if (!token || !me) return 登录页`，所以会短暂显示登录页。
- 服务器重启后仍不用重新输入账号密码，是因为浏览器里保存的 JWT 仍有效；这本身不是 bug。真正的问题是校验期间 UI 闪烁，以及 token 失效时清理逻辑不够清晰。

改法：

1. 管理端 auth 状态改为三态：`checking`、`authenticated`、`anonymous`。
2. 有 token 但还没完成 `/api/auth/me` 前，展示全屏加载状态，不展示登录页。
3. 只有 `/api/auth/me` 返回 401/403 时才清 token 并进入登录页；网络错误、服务器短暂重启、502 不立即清 token，而是展示“连接失败，请稍后重试”。
4. 登录成功后立刻设置 `me`，再进入后台。
5. 如果需要“每次服务器重新部署都强制重新登录”，应通过更换 JWT secret 或服务端 token 黑名单实现，但这会影响体验，当前不建议作为默认行为。

验收：

- F5 刷新后台页面不会闪登录页。
- 后端短暂重启时不自动退出，只显示连接失败或重试。
- JWT 过期或账号被禁用时才进入登录页。

## 3. P1 重要体验修复

### P1-1 渠道最近成功时间精确到分钟

目标：管理端渠道列表和详情页时间简洁可读。

改法：

1. 增加统一时间格式工具：`formatDateMinute(value)`。
2. 输出格式建议：`2026-06-18 18:42`。
3. 渠道列表 `最近成功`、渠道详情 `最近成功`、最近错误时间统一使用该工具。

验收：

- 不再显示秒、毫秒、时区长后缀。
- 空值显示 `暂无` 或 `-`。

### P1-2 Anthropic 协议检测与 `/v1` 说明修正

目标：用户选择 Anthropic 时，检测逻辑优先按 Anthropic 走；界面不再让人误以为系统错误追加 `/v1`。

当前逻辑说明：

- Anthropic 标准 SDK 习惯是用户填写不带 `/v1` 的 `base_url`。
- 网关真正请求 Anthropic 时必须访问 `<base_url>/v1/models` 和 `<base_url>/v1/messages`。
- 当前 `anthropicURL()` 会在内部自动拼 `/v1`，如果用户已经填了 `/v1` 则跳过拼接，这是合理的。
- 问题在检测接口没有接收用户当前选择的协议。自定义域名不包含 `anthropic` 或 `claude` 时，当前会先试 OpenAI，失败后才试 Anthropic；如果 OpenAI 的 `<base>/v1/models` 刚好返回成功，就会被识别为 OpenAI。

改法：

1. `POST /api/admin/channels/detect` 增加可选 `provider_type`。
2. 前端点击“检测并填充协议”时，把当前表单里的协议一并传给后端。
3. 后端规则：
   - `provider_type=anthropic`：先用严格 `ProbeAnthropic`。
   - `provider_type=openai`：先用严格 `ProbeOpenAI`，再尝试 OpenAI `/v1`。
   - 未传协议：保持自动识别，但返回探测过程摘要。
4. UI 增加两行结果：
   - `保存地址`：写入数据库的 base_url。
   - `实际探测地址`：例如 Anthropic 显示 `<base_url>/v1/models`。
5. 不复用带预设回退的 `AnthropicAdapter.ListModels` 做探测，继续使用严格探针。

验收：

- 选择 Anthropic 后检测不会错误填成 OpenAI。
- 用户填写 `https://api.anthropic.com` 时保存地址不强制变成 `/v1`，但结果里明确显示实际探测了 `/v1/models`。
- 用户填写已带 `/v1` 的历史地址仍兼容。

### P1-3 写操作后数据实时刷新

目标：新增、删除、编辑、检测、导入后，当前页面立即反映最新状态。

改法：

1. 梳理每个页面的数据归属：列表数据、详情数据、弹窗局部数据分开刷新。
2. 写操作成功后执行明确的刷新动作：
   - 列表页操作：刷新当前列表和相关统计。
   - 详情页操作：刷新详情，不依赖返回上一级再回来。
   - 弹窗操作：关闭弹窗后刷新所在页面数据。
3. 管理端保留轻量方案，不引入大型状态库。建议封装 `useReloadableResource` 或 `reloadAdminShell()`，避免继续在 `main.tsx` 堆所有状态。
4. 用户端在创建/删除 API Key、兑换成功、修改设置后，立即刷新对应列表和 dashboard。
5. 页面重新获得焦点时，对余额、dashboard、渠道健康、列表页做一次轻量重拉。

验收：

- 删除渠道、删除模型、停用密钥、导入模型、兑换码禁用后，无需手动刷新即可看到变化。
- 用户兑换成功后余额立即变化。
- 用户删除 API Key 后列表立即变化。

### P1-4 前后端错误体解析统一

目标：错误提示展示真实中文信息，不再出现 `[object Object]` 或泛化错误。

改法：

1. 前端 API client 统一解析以下格式：
   - `{ "error": "文本" }`
   - `{ "error": { "message": "文本", "code": "xxx" } }`
   - `{ "message": "文本" }`
2. 401/403/429 的全局处理保留，但要避免同一错误既触发全局提示又触发页面提示。
3. 后端 `httpx.Error` 输出结构统一，不破坏 v1.4 上游错误透传。

验收：

- 上游余额不足、渠道检测失败、权限不足、参数错误都能显示真实中文或上游 message。
- 401 只提示一次。

## 4. P2 顺手收口：不紧急但建议一起做

### P2-1 全量中文乱码清理

当前管理端存在大量乱码文案，例如菜单、按钮、表格标题、Modal 标题。这会显著降低可信度。

改法：

1. 全量扫描前后端中文乱码，优先修管理端 `frontend/admin/src/main.tsx` 和 `frontend/admin/src/pages/*`。
2. 所有面向用户/管理员的英文操作词统一为中文，协议名、Base URL、API Key、RPM 等行业固定词保留。
3. 增加简单脚本或检查命令，扫描常见乱码片段，防止再次提交。

验收：

- 用户端和管理端没有中英混杂的普通文案。
- 管理端菜单、弹窗、表单、表格标题全部可读。

### P2-2 用户端敏感字段彻底剥离

当前用户侧共享类型没有倍率字段，但后端用户门户内部结构仍包含 `RateMultiplier`。建议从用户响应结构的源头移除。

改法：

1. `UserModelPrice` 去掉 JSON 可导出的 `rate_multiplier`、基准成本字段。
2. DTO 只输出用户应付单价：输入单价、输出单价、单次价格。
3. 公共价格表和用户模型接口增加脱敏测试，断言响应体不包含 `base_cost`、`rate_multiplier`、`gross_profit`。

验收：

- `/api/user/models`、`/api/public/models` 响应体均不包含敏感字段。
- 管理端仍能看到成本、倍率、毛利。

### P2-3 管理端结构瘦身

当前 `frontend/admin/src/main.tsx` 承担了菜单、鉴权、全局数据、列定义、弹窗、页面路由等职责，后续越改越容易互相影响。

改法：

1. 抽出 `AdminAuthProvider`。
2. 抽出 `AdminShell`、`AdminMenu`、`AdminRoutes`。
3. 表格列定义下沉到对应页面，避免主文件继续膨胀。
4. 只做结构拆分，不重做 UI，不改变功能。

验收：

- 主入口文件明显变薄。
- 原页面功能不回退。

## 5. 与 sub2api 的差距和轻量优化建议

你当前项目已经不是简单 sub2api 克隆，而是带余额、倍率计费、账本、管理端、用户端的私有运营网关。和 sub2api 类项目相比，建议只补“中转站核心体验”，不要做成大而全平台。

建议补齐的轻量能力：

1. 渠道体验：供应商预设、协议探测、模型同步、渠道健康、错误透传要继续打磨，这是中转站最核心的运营效率。
2. 模型价格模板：导入模型后提供常见模型价格参考，但最终仍由管理员确认，避免自动定价出错。
3. SDK 接入指引：用户端提供 OpenAI SDK、Anthropic SDK 的最短示例，只展示平台 base_url 和 key 使用方式，不展示倍率和扣费公式。
4. 健康视图：管理端显示渠道可用性、最近失败原因、最近成功时间、绑定模型数，足够定位问题即可。
5. 账务可信：余额、冻结、扣费记录、调用日志必须准确；这是比花哨功能更重要的护城河。

不建议做的能力：

1. 在线支付、分销、代理层级、工单系统。
2. 过重的 BI 大屏、复杂增长工具。
3. 面向普通用户展示成本、倍率、毛利、上游真实采购结构。
4. 过度复杂的权限系统；当前管理员/普通用户足够。

## 6. 建议执行顺序

验收本方案后，建议按以下顺序实施：

1. P0-1 统一提示系统。
2. P0-2 修 API Key 老数据删除。
3. P0-3 修管理端刷新闪登录。
4. P1-1 时间格式化。
5. P1-2 Anthropic 检测携带协议提示。
6. P1-3 写操作后实时刷新。
7. P1-4 错误体解析统一。
8. P2-1 中文乱码清理。
9. P2-2 用户端敏感字段剥离。
10. P2-3 管理端结构瘦身。

每个小节完成后都跑：

```powershell
cd D:\code\LingShu\backend
go build ./...
go vet ./...
go test ./...

cd D:\code\LingShu\frontend
npm --workspace @lingshu/user run build
npm --workspace @lingshu/admin run build
```

## 7. 最终验收清单

- 用户端、管理端同一操作只出现一次提示。
- 用户端老 API Key 和新 API Key 都能正常删除，删除后立即从列表消失。
- 渠道最近成功时间显示到分钟。
- Anthropic 检测逻辑能解释并正确处理 `/v1`，选择 Anthropic 时不会误判 OpenAI。
- 管理端刷新不会闪登录页，网络短暂异常不会误清登录状态。
- 常见写操作无需手动刷新页面即可看到结果。
- 用户端接口不暴露 `base_cost`、`rate_multiplier`、`gross_profit`。
- 管理端中文文案无乱码。
- 全套后端 build/vet/test 和两个前端 build 通过。
