# 灵枢 v1.1 上线返工方案

> 目标：把已上线版本从“能跑”修成“可运营、可理解、可扩展”。
> 本文只做返工方案，不重新推翻 v1.0 的计费内核与数据库 schema。

## 1. 背景

当前线上版本已经具备基础链路，但还存在五类明显问题：

1. 用户端与管理端中英混用、部分文案乱码、枚举值直接裸显。
2. 用户端暴露了不该给用户看的内部概念，例如 JWT、倍率、成本、扣费公式、可追溯、充值方式、Postgres/Redis 等实现细节。**更严重的是后端 `/api/user/*` 接口响应体里仍带着 `base_cost`、`rate_multiplier`、`gross_profit`，用户拿自己的 JWT 直接 curl 就能算出进价和毛利。**
3. 管理端导航与内容区结构混乱，左侧菜单不可用或与仪表盘功能冲突。整个管理端单文件 494 行堆砌，无路由、无分页。
4. CRUD 不完整，模型、渠道、公告、兑换码、用户、API Key 都缺少常规编辑/删除/禁用入口。用户详情页缺失导致调余额时无法查看用户消费明细。
5. 上游兼容性仍偏 OpenAI-only，需要扩展 Anthropic 等 provider。
6. 缺少全局错误处理、Toast 反馈、Error Boundary、空状态中文化。

## 2. 返工原则

1. 计费语义不变：`base_cost × rate_multiplier = charge`，仍然是平台核心。
2. 用户端面向客户，必须只展示客户需要理解的信息。
3. 管理端面向运营，强调效率、审计和可批量操作。
4. 任何新增 provider 都必须通过统一适配层接入，不允许把分支逻辑散在各个 handler 里。
5. 本次只做文档内范围，不引入支付网关、分销、多级权限、工单系统。

## 3. 推荐架构

### 3.1 前端

- 保留双前端：`frontend/user` 与 `frontend/admin`。
- 用户端继续使用 `shadcn/ui + Tailwind`，目标是偏产品化、精致、克制。
- 管理端继续使用 `Ant Design`，目标是高密度、高效率、适合运营。
- 两端共用 `frontend/packages/shared` 的设计 token、API client，但**类型定义需拆分为 `AdminTypes` 和 `UserTypes`**：
  - `UserTypes` 去掉 `base_cost`、`rate_multiplier`、`gross_profit` 等敏感字段。
  - `AdminTypes` 保留完整字段用于管理端。
  - 后端对应建 `internal/dto` 层，user 接口序列化时用 `UserLedgerRecordDTO` 等，admin 接口序列化时用完整类型。

### 3.2 后端

- 保持现有业务分层。
- 在 `internal/upstream` 下补 provider adapter：
  - `openai_adapter`
  - `anthropic_adapter`
  - `openai_compatible_adapter`
- 统一使用内部标准请求/响应结构，再分别做 provider 转换。

## 4. 返工分期

### P0：先让线上可用、可看、可操作

#### 范围

- **后端安全收口（最高优先级）**：
  - **剥离 user-scoped 接口响应中的敏感字段**：`/api/user/*` 下所有接口的 JSON 响应里去掉 `base_cost`、`rate_multiplier`、`gross_profit`，在序列化层（或专门的 user DTO）做过滤，确保用户即便直接 curl 也看不到进价和毛利。
  - 涉及接口：`/api/user/dashboard`、`/api/user/usage/logs`、`/api/user/usage/ledger`、`/api/user/models`。
  - 前端 `packages/shared/src/types.ts` 对应拆分为 `AdminTypes` 和 `UserTypes`。
  
- **全站中文化**：
  - 清理所有中英混用：导航、按钮、表格列名、表单 label、placeholder、toast、空状态、错误提示。
  - 枚举值映射：`active/disabled/enabled` → `启用/禁用`，`per_call/token` → `按次计费/按量计费`，`success/failed` → `成功/失败`，`redeem/charge/refund` → `兑换/扣费/退款`。
  - 配置 AntD `ConfigProvider locale={zhCN}`。
  
- **用户端脱敏**：
  - 删除所有内部运营词汇：JWT、倍率、成本、扣费公式（`base_cost × rate_multiplier`）、可追溯、充值方式细节、Postgres、Redis、gateway_requests、行锁、`/api/auth/change-password` 等实现细节。
  - 登录页英文标语（`”A calm cockpit for private AI traffic.”`、`”追踪扣费和倍率毛利链路”`）全部改为中文且去掉内部概念。
  - `models.tsx` 模型卡片不再显示 `rate_multiplier`，`settings.tsx` 不再显示扣费公式。
  
- **管理端重构导航**：
  - 用 `react-router-dom` 建立真实路由，拆出 `pages/` 和 `layouts/`。
  - 左侧 `MenuPlaceholder` 换成真正的 AntD `<Menu>` + `onClick` 驱动路由跳转。
  - 去掉与左侧菜单功能重叠的顶部统计卡（仪表盘冲突问题）。
  - 拆分单文件（`main.tsx` 494 行）为多页面结构。
  
- **补齐基础 CRUD**：
  - **用户**：基础详情页（点击用户名跳转，显示余额、今日消费、API Key 列表、最近 20 条账本、最近 20 条请求日志）、编辑、禁用、重置密码、调余额。
  - **模型**：编辑、删除、上下架。
  - **渠道**：编辑、删除、测试、**解绑模型**、健康状态重置。
  - **公告**：编辑、删除、置顶。
  - **兑换码**：禁用、导出。
  - **API Key**：禁用、删除、重命名、**重新启用**。
  
- **全局反馈与错误处理**：
  - 用户端：添加 shadcn `toast()` + `<Toaster />` 组件，所有异步操作（创建/删除/禁用/兑换/登录）都用 toast 反馈成功/失败。
  - 管理端：统一用 AntD `message.success() / message.error()`，删除现有的 Alert 一次性提示。
  - 两端都加 Error Boundary（捕获组件错误显示友好提示 + 上报到审计日志）。
  - 表格/列表空状态全部中文化，提供引导操作（例如”暂无模型，点击上方「创建模型」开始配置”）。
  
- **分页支持**：
  - 管理端所有列表（用户/模型/渠道/日志/账本/审计/公告/兑换码）必须支持分页，单页默认 20 条。
  - 后端接口支持 `?page=1&limit=20`，返回 `{ items: [], total: 100, page: 1, limit: 20 }`。

#### 验收

- **安全收口**：用户端所有接口响应（包括直接 curl）都不再返回 `base_cost`、`rate_multiplier`、`gross_profit`。
- **中文化**：两端不再出现英文 UI 文案（除品牌名、API endpoint），枚举值全部有中文映射。
- **脱敏**：用户端不再出现内部运营词（倍率/成本/公式/JWT/Postgres/Redis/可追溯/充值方式等）。
- **管理端导航**：左侧菜单全部可点击，且与主内容不冲突，路由切换正常。
- **CRUD 完整性**：至少一套完整 CRUD 链路（创建→编辑→禁用→删除→查看详情）可运行，用户详情页显示余额和消费明细。
- **反馈机制**：所有异步操作都有 toast/message 成功/失败反馈，表格空状态有中文引导。
- **分页**：管理端所有列表支持分页，单页 20 条，后端接口返回 `{ items, total, page, limit }`。
- **危险操作**：删除、禁用有确认弹窗和审计日志。

### P1：补齐运营闭环

#### 范围

- **管理端概览重做**：收入、成本、毛利、活跃用户、请求量、异常率。
- **用户详情页增强**（基础版在 P0 完成）：
  - 增加筛选条件（按日期范围、按模型、按状态）。
  - 账本和请求日志支持导出 CSV。
  - 显示用户创建时间、最后活跃时间、累计消费、累计充值。
- **模型详情页**：公开模型名、上游模型、计费方式、倍率、状态、绑定渠道列表、累计调用量。
- **渠道详情页**：权重、健康、失败次数、RPM、并发、绑定模型列表、累计请求量、平均延迟。
- **报表拆分**：按日、按用户、按模型、按渠道的聚合报表，支持导出。
- **审计日志增强**：支持筛选（操作者、动作、对象类型、时间范围）。
- **网络错误统一处理**：401/403/429/500 等状态码统一拦截，显示对应中文提示（如"登录已过期，请重新登录"、"请求过于频繁，请稍后再试"）。

#### 验收

- 管理员可以完成日常运营，不需要碰数据库。
- 所有核心资源都能查、能改、能禁用、能追踪。

### P2：模型兼容扩展

#### 范围

- 增加 Anthropic provider。
- 补齐其他 OpenAI 兼容上游的 adapter。
- 保持用户侧调用方式不变，仍然是平台统一 OpenAI SDK 接入。
- 统一处理：
  - 请求转换
  - 响应转换
  - 流式 SSE
  - usage 提取
  - 错误映射

#### 验收

- 同一平台 API Key 可调用 OpenAI-compatible 与 Anthropic 类模型。
- 请求日志、账本、余额扣费仍然一致。

### P3：稳定性与收口

#### 范围

- 统一文案与组件风格。
- 补单元测试与集成测试。
- 补弱网、限流、失败切换和边界错误提示。
- 固化部署与 README。

#### 验收

- `go build ./...`、`go test ./...` 通过。
- 前端构建通过。
- 线上截图和关键路径可复现。

## 5. 不做什么

- 不做在线支付。
- 不做分销和多级权限。
- 不做工单系统。
- 不做聊天产品本体。
- 不改变现有计费语义。

## 6. 建议实施顺序

1. 先做 P0，尤其是中文化、脱敏和管理端导航。
2. 再补 P1，把运营闭环补完整。
3. 最后做 P2，把 provider adapter 抽干净。
4. P3 只做收口，不继续扩功能。

## 7. 风险点

- **文案返工容易漏字段**：尤其是表格列名、空状态、错误提示、枚举值。建议执行时用 `grep -r "token\|status\|type\|active\|disabled" frontend/` 全局扫一遍确保都有中文映射。
- **后端剥敏与前端类型对齐**：剥离敏感字段后，前端 TS 类型定义必须同步拆分，否则类型和实际响应对不上会导致运行时错误。
- **管理端导航重构时容易和现有数据加载逻辑冲突**：现在所有数据在 App 组件 `useEffect` 里一次性 `Promise.all` 加载，改成路由后需要按页面懒加载，避免首屏加载 13 个接口。
- **provider adapter 改造时容易把计费和转发耦合回去**：必须保持分层，adapter 只负责协议转换，计费逻辑仍在 `billing` 和 `service` 层。
- **用户端脱敏时要避免把有用信息一起删掉**：重点是不展示内部成本和倍率，但用户自己的消费金额、余额、请求状态等必须保留且清晰展示。
- **分页改造时注意无限滚动 vs 传统分页器的选择**：管理端适合传统分页器（AntD Table 自带），用户端的日志/账本列表可考虑无限滚动（shadcn 没现成组件，需自行实现或用传统分页）。

## 8. 结论

本次 v1.1 的核心不是”继续加功能”，而是把平台修成：

- **用户看得懂**：中文化、脱敏、去掉所有内部概念。
- **管理员用得顺**：真实路由、完整 CRUD、分页、详情页、筛选导出。
- **模型接得进**：provider adapter 统一适配层。
- **计费算得清**：已在 v1.0 验收通过，本次重点是不泄露成本和毛利。
