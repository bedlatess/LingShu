## 1. 项目定位

灵枢是一个面向**私有运营**的 AI API 聚合网关。它把多个上游 AI 服务（OpenAI / Claude / Gemini / 各类 OpenAI 兼容的国产模型）的额度聚合到统一入口，向下游用户发放平台 API Key。平台负责：鉴权、余额校验、请求转发、用量记录、**按倍率扣费**、报表展示、公告与兑换码。

**商业模式核心：** 平台以「上游真实成本 × 自定义倍率」从用户**余额**中扣费，倍率差价即平台利润。

## 2. 核心原则（必读，全程遵守）

1. **精简优先**：只做第 3 节范围内的功能，不擅自添加 sub2api 的臃肿功能（多支付网关、多级代理、复杂租户、营销系统）。
2. **计费即生命线**：第 7 节的「倍率 + 余额 + 预占结算」是全系统核心，必须精确、并发安全、可对账、可算毛利。**每一分钱的变动都能追溯到调用记录、管理员操作或兑换码。**
3. **用户端要精致，管理端要高效，二者视觉语言不同**（第 9 节）。**禁止**把用户端做成参考截图里那种「裸按钮竖排 + 密集表格」的开发者工具样式。
4. **OpenAI 兼容为第一优先级**：统一对外暴露 OpenAI 兼容接口，客户端零改造接入。
5. **先闭环再扩展**：第一版跑通「登录 → 配置模型/渠道 → 发 Key → 网关转发 → 倍率扣费 → 记录 → 报表 → 公告 → 兑换码」。
6. **角色仅两种**：管理员、普通用户。无任何上下级 / 分销 / 子账号。
7. **合规**：系统只作技术平台，不内置任何绕过上游服务条款的机制；上游资源是否允许该用途由运营者自行确认。

## 4. 明确不做（第一版）

- ❌ 任何在线支付网关（支付宝/微信/Stripe/EasyPay）——充值只通过**兑换码** + **管理员手动充值**
- ❌ 多级代理 / 分销 / 上下级返佣 / 子账号
- ❌ 多级角色权限（只有 admin + user）
- ❌ 团队 / 组织 / 租户体系
- ❌ 用户自助实名认证
- ❌ 聊天应用本体（只提供 API 网关 + 用量查询）
- ❌ OAuth / 订阅账号类复杂上游（第一版只做标准 API Key 上游）
- ❌ 工单系统（售后只在公告里放联系方式/二维码）
- ❌ TOTP/2FA、Turnstile（保持精简，可放后续）

## 5. 技术栈

> 默认推荐如下。开发者若更熟悉同级方案可微调，但**必须**保证：前后端分离、关系型数据库 + Redis、流式转发性能、并发扣费原子性。

### 后端
| 项 | 选型 | 说明 |
|----|------|------|
| 语言 | **Go 1.22+** | 高并发转发友好 |
| Web 框架 | **Gin** 或 **Chi** | 二选一；Chi 更标准库风格，Gin 生态更熟，均可 |
| DB 访问 | **sqlc**（推荐）或 GORM | **账本/报表 SQL 用 sqlc 更可控、防错**；若团队偏好 ORM 可用 GORM，但计费相关 SQL 建议手写 |
| 主存储 | **PostgreSQL 15+** | 业务数据、账本、调用日志 |
| 缓存/并发 | **Redis 7+** | 余额预占冻结、并发计数、限流、会话粘性、热点统计 |

### 前端
| 项 | 选型 | 说明 |
|----|------|------|
| 框架 | **React + TypeScript + Vite** | |
| **用户端 UI** | **shadcn/ui + TailwindCSS** | **追求精致**，Vercel/Linear 风格，见第 9 节 |
| **管理端 UI** | **Ant Design 5** | 追求**操作效率**，表格/筛选/表单密度高 |
| 数据请求 | **TanStack Query** | 统一缓存与刷新 |
| 路由 | **React Router** | 区分用户端 / 管理端 |
| 图表 | **ECharts**（管理端复杂报表）/ **Recharts**（用户端轻量趋势） | |

> **为何用户端与管理端用不同组件库？** 用户端是面向客户的「产品」，要精致、有品牌感；管理端是面向自己的「工具」，要高效、信息密。Ant Design 做精致产品偏模板感，shadcn 做密集后台偏重。各取所长。两端共用一套设计 token（主色、圆角、间距）保证品牌统一。

### 部署
- **Docker Compose** 单机一键启动：app + postgres + redis
- Nginx / Caddy 负责 HTTPS 与反代（注意 `underscores_in_headers on;` 以支持会话粘性头）
- 后端全环境变量配置；PG/Redis 持久化挂载

## 6. 系统架构

### 6.1 数据流

```text
                       ┌──────────────────────────────────────────────┐
  下游客户端            │                    灵枢网关                     │      上游 AI 提供商
 (OpenAI SDK 等)        │                                                │
      │                │  ┌────────┐   ┌──────────────────────────────┐ │     ┌─────────────┐
      │  Bearer sk-xxx │  │ 鉴权    │   │  计费引擎                     │ │     │ OpenAI      │
      ├───────────────▶│  │ APIKey │──▶│  预占 → 选渠道 → 转发 → 真实结算 │─┼────▶│ Claude      │
      │                │  └────────┘   └──────────────────────────────┘ │     │ Gemini      │
      │◀──流式/非流式───│       │                      │                 │     │ 国产兼容模型 │
      │                │       ▼                      ▼                 │     └─────────────┘
                       │   ┌───────┐            ┌──────────┐            │
                       │   │ Redis │            │ Postgres │            │
                       │   │预占/并发│            │账本/日志  │            │
                       │   └───────┘            └──────────┘            │
                       └──────────────────────────────────────────────┘
            ▲                                                ▲
            │ JWT                                            │ JWT (role=admin)
     ┌──────────────┐                                ┌──────────────┐
     │ 用户端 (shadcn)│                                │ 管理端 (AntD) │
     └──────────────┘                                └──────────────┘
```

后端同时提供两类能力：
1. **业务 API**（`/api/...`）：登录、用户、报表、公告、兑换码、配置（JWT 鉴权）
2. **网关 API**（`/v1/...`）：OpenAI 兼容入口，负责转发与计费（API Key 鉴权）

### 6.2 上游渠道与调度（精简版）

- 一个**模型**可由多个**上游渠道**提供（多对多，经 `channel_models` 关联表，支持模型名映射）。
- 渠道字段：类型、base URL、加密 API Key、状态、权重、超时、RPM/并发限制、熔断阈值、健康状态、最近成功/失败时间、最近错误。
- **选渠道策略**：
  1. 过滤「启用 + 健康 + 支持该模型 + 未达并发上限」的候选。
  2. 候选内**按权重轮询/加权随机**。
  3. **会话粘性**（默认开）：相同会话标识（请求头 `X-Session-Id`，缺省则 `apiKey+model` 哈希）在 TTL 内尽量命中同一渠道。
- **失败处理**：转发失败自动重试下一候选（最多 `max_retry` 次）；渠道连续失败超阈值 → 标记 `unhealthy` 熔断，冷却后自动探测恢复。

### 6.3 网关对外接口（OpenAI 兼容）

| 路径 | 方法 | 第一版 | 说明 |
|------|------|:----:|------|
| `/v1/models` | GET | ✅ | 返回该 Key 可用模型（OpenAI 格式） |
| `/v1/chat/completions` | POST | ✅ | 聊天补全，支持 `stream:true` SSE |
| `/v1/embeddings` | POST | 可选 | 向量 |
| `/v1/images/generations` | POST | 可选 | 文生图（按次计费） |
| `/v1/completions` | POST | 可选 | 传统补全 |

鉴权：`Authorization: Bearer <平台key>`。
流式：正确透传 SSE；流结束后用上游 `usage` 做真实结算，缺失时 tokenizer 估算并标记 `is_estimated`。

## 7. 计费引擎（系统核心，重点实现）⭐

> **这是本项目区别于参考稿的最关键修正。** 计费 = sub2api 的 token 级核算 **× 自定义倍率**，从**用户余额**扣费。**不使用「直接定单价」，更不使用「按次积分」。** 必须能算出毛利。

### 7.1 核心概念

| 概念 | 说明 |
|------|------|
| **余额 balance** | 用户账户里的额度，高精度 `DECIMAL(18,6)` 存储。对外称「余额」，内部即点数。 |
| **基准成本 base_cost** | 按上游真实 token 用量 × 模型**基准单价**算出，代表「1 倍」上游成本。 |
| **倍率 rate_multiplier** | 管理员为每个模型设置的加价系数（如 1.200、1.300）。默认取系统默认倍率，可单独覆盖。 |
| **实际扣费 charge** | `charge = base_cost × rate_multiplier`。**charge − base_cost 即平台毛利。** |

### 7.2 模型价格字段（关键，区别于参考稿）

每个模型维护：

| 字段 | 含义 |
|------|------|
| `billing_mode` | `token`（按 token）或 `per_call`（按次，如文生图） |
| `input_price_per_1k` | 每 1K 输入 token 的**基准成本**（1 倍，对应你付给上游的价） |
| `output_price_per_1k` | 每 1K 输出 token 的**基准成本**（1 倍） |
| `price_per_call` | 按次模式的**基准成本**（1 倍） |
| `rate_multiplier` | 该模型倍率，如 1.300 |

> **设计要点：录入的是「上游成本」而非「对外售价」。对外售价 = 成本 × 倍率，由系统实时计算。** 这样调利润只需改倍率，且报表能算毛利。

### 7.3 计费公式

```text
# token 模式
base_cost = (input_tokens  / 1000) * input_price_per_1k
          + (output_tokens / 1000) * output_price_per_1k
charge    = base_cost * rate_multiplier

# 按次模式（文生图等）
base_cost = price_per_call * n           # n = 图片数量
charge    = base_cost * rate_multiplier
```

精度：DB 用 `DECIMAL/NUMERIC`（禁用 float）；内部计算保留 8 位小数；扣费 charge 向上取整到 6 位小数；前端展示 3~6 位。

### 7.4 预占 → 转发 → 真实结算（并发安全，重点！）

对应截图「**请求中预占**」。流程：

```text
1. 鉴权通过，解析目标 model，校验模型启用 + 用户/Key 状态。
2. 预估上限 estimate = (输入token实算 + max_tokens 上限) 按 7.3 折算的 charge。
3. 【Redis 原子】检查并冻结：
     if balance - frozen < estimate  →  拒绝 402 余额不足
     else  frozen += estimate
4. 选上游渠道（6.2），转发请求（流式/非流式），失败重试下一候选。
5. 拿到真实 usage(input_tokens, output_tokens)：
     base_cost、charge 按 7.3 计算。
6. 【Postgres 事务 + 行锁 SELECT...FOR UPDATE】结算：
     balance -= charge
     写 balance_ledger（记 base_cost、rate_multiplier、charge）
     写 gateway_requests（调用明细）
   【Redis】frozen -= estimate   # 释放预占
7. 失败/超时/中断：释放全部预占，不扣费（若上游已产生可计费用量，按已产生部分结算，状态记 partial）。
```

- 余额扣费 + 账本写入 **同一事务**，保证 `Σledger.amount == 充值总额 − 当前余额`。
- 预占用 Redis（实时、高频、防超额），最终账用 Postgres（持久、可对账）。

### 7.5 token 计数

- 优先用上游响应 `usage`。
- 流式且上游无 usage：用 tokenizer（tiktoken Go 移植等）估算 prompt + 累计输出，记 `is_estimated=true`，模型配置标注 tokenizer 类型。

## 8. 产品模块详述

### 8.1 认证与账号
- 用户名/邮箱 + 密码登录；JWT 登录态（`jwt.expire_hour` 默认 24h）。
- 密码 Argon2id（推荐）或 bcrypt(cost≥10)。
- 用户状态：active / banned；角色：admin / user。
- 管理员可建用户、重置密码、封禁。自助注册由 `registration_enabled` 开关控制（默认关）。
- 登录失败需限流（防爆破）。

### 8.2 用户 API Key
- 每用户可多 Key；Key 仅创建时显示明文一次，之后只显示脱敏（`lsk_live_...HFCk`）。
- DB 只存哈希 + 前缀；日志只记脱敏值；**管理员也无法查看已建 Key 明文**。
- 字段：状态、备注名（如截图「贝贝南瓜」）、过期时间、请求限制。
- 鉴权：`Authorization: Bearer <key>`。
- Key 格式建议：`lsk_live_` + 32 位随机。

### 8.3 模型广场与价格
- 用户端展示：显示名、类型、计费方式、**该用户实际单价（基准 × 倍率）**、状态。
- 模型类型：`chat`（按 token）/ `embedding`（按 token）/ `image`（按次）/ `video`（预留）。
- 模型可映射到不同上游真实模型名（经 `channel_models.upstream_model_name`）。

### 8.4 上游渠道管理
- 配置 base URL、加密 API Key、类型、状态、权重、超时、RPM/并发、熔断阈值。
- 渠道绑定支持的模型。
- 管理端展示健康状态、最近失败原因、最近成功时间，提供**测试连通**按钮（仅返回成功/错误摘要，不回显 Key）。

### 8.5 网关转发
- 流程见 7.4。失败码：401 鉴权失败 / 402 余额不足 / 404 模型不存在 / 429 限流 / 502 上游全失败。
- 所有失败写日志但不扣费（除非已产生可计费用量）。
- 流式：SSE 透传，结束后补写日志与账本；中断记 partial。

### 8.6 余额与账本
- 统一余额单位（内部 decimal）。**余额变动必须经账本，禁止裸改余额。**
- 账本类型：`admin_grant` / `admin_deduct` / `redeem` / `usage_charge` / `refund` / `adjustment`。
- 每条记录：用户、变动额、变动前余额、变动后余额、类型、关联对象 ID、操作人、备注、时间。
- **usage_charge 类型额外记 `base_cost`、`rate_multiplier`、`charge`**（算毛利用）。

### 8.7 兑换码
- 管理员单个/批量生成；可设面额、过期、可用次数（推荐单次）、批次名、备注。
- 状态：unused / used / expired / disabled。
- **兑换原子性**：`UPDATE ... WHERE status='unused'` + 事务，核销与加余额同成同败。
- 记录兑换人、时间、IP（独立 `redeem_records` 表）。

### 8.8 公告
- 管理员增删改、上下线、置顶、设发布/过期时间。
- 正文 Markdown（安全渲染，防 XSS）；可放图片 URL、售后二维码/联系方式。
- 不做富文本编辑器。

### 8.9 审计日志（吸收自参考稿，保留）
- 记录范围：建/禁/删用户、重置密码、改余额、建/禁 Key、增改删模型、增改禁渠道、生成/禁兑换码、发布/删公告、改系统设置。

**布局**：经典「左侧可折叠菜单 + 顶部面包屑 + 主区表格/表单」。这里**允许**信息密集，因为是运营工具。

**菜单项**：仪表盘 / 用户 / 模型 / 渠道 / 调用日志 / 账本 / 兑换码 / 公告 / 设置 / 审计。

## 13. 安全设计

- **平台 API Key**：只存哈希，只展示前后缀脱敏，可禁用，日志不记完整 Key，管理员也看不到明文。
- **上游 Key**：AES-GCM 加密存储（密钥来自 `KEY_ENCRYPTION_SECRET` 环境变量），前端永不回显明文，测试渠道只返回成功/错误摘要。
- **金额安全**：所有变动走事务 + 行锁；兑换码原子核销；余额扣费防并发扣负。
- **管理端**：所有接口校验 `role=admin`；危险操作写审计 + 必填备注；登录失败限流。
- **网关**：请求体大小限制；单 Key 并发限制；单 Key RPM 限制（Redis 令牌桶）；可选单用户每日消耗上限；上游错误不暴露内部配置/堆栈。
- **通用**：SQL 参数化防注入；Markdown 安全渲染防 XSS；敏感数据不入 URL query。

## 15. 风险与处理（吸收自参考稿）

| # | 风险 | 处理 |
|---|------|------|
| 1 | 上游 usage 不稳定/缺失 | 优先用上游 usage；缺失则 tokenizer 估算，记 `is_estimated` |
| 2 | 流式扣费延迟 | 先透传响应，结束后写日志+账本；中断记 partial 状态 |
| 3 | 余额并发扣负 | Redis 预占冻结 + Postgres 事务行锁；不足即拒 |
| 4 | 渠道失效/额度耗尽 | 渠道测试 + 失败计数 + 熔断 + 管理端告警标记 + 自动切换 |
| 5 | 倍率/价格改动影响在途请求 | 计费以请求**发起时**快照的价格与倍率为准，写入该次 gateway_requests，事后改价不影响历史账 |
| 6 | 合规 | 仅作技术平台，不内置绕过机制；上游用途由运营者自负 |

## 16. 开发阶段与验收

> 严格按序推进，每阶段达标再进下一阶段。

### 阶段 0 · 项目骨架
- 后端/前端/Docker Compose 初始化；接入 PG + Redis；配置、日志、迁移系统。
- **验收**：一键启动；`/healthz` 可访问；前端空壳可访问；迁移可执行。

### 阶段 1 · 认证与基础后台
- 登录、角色权限、初始管理员、用户管理、审计日志基础。
- **验收**：管理员能登录管理端并建普通用户；普通用户无法访问 admin API；关键操作有审计。

### 阶段 2 · 模型/渠道/Key 管理
- 模型管理（**含基准价 + 倍率**）、上游渠道管理、渠道测试、API Key 创建/禁用/哈希存储。
- **验收**：能配一个 OpenAI 兼容上游；能配模型价格与倍率；渠道测试显示成功/错误；用户 Key 可用于后续鉴权。

### 阶段 3 · 网关转发与倍率扣费 ⭐（最关键）
- `/v1/models`、`/v1/chat/completions`（非流式 → 流式）；预占→选渠道→转发→真实结算；写日志+账本。
- **验收**：
  - [ ] OpenAI SDK 改 base_url+key 可正常调用（含流式）
  - [ ] 余额不足被拒 402；充足正常扣费
  - [ ] **并发 50 路同时请求，账本无超扣，Σ扣费与余额变动一致**
  - [ ] 倍率改 1.3 后新请求 charge = base_cost × 1.3
  - [ ] 上游某渠道故障自动切换到健康渠道并成功返回
  - [ ] 日志同时记录 base_cost / rate_multiplier / charge

### 阶段 4 · 兑换码与公告
- 管理端生成/批量兑换码、公告管理；用户端兑换、公告展示。
- **验收**：兑换成功加余额且记录；重复/过期兑换失败；管理员发公告用户可见；兑换记录与账本一致。

### 阶段 5 · 报表与日志完善
- 用户端 5 张报表；管理端仪表盘 + 全站日志/账本筛选 + **毛利报表**。
- **验收**：用户看懂自己用量与扣费；管理员可按用户/模型/时间筛选；**毛利 = Σcharge − Σbase_cost 正确**；每个用户余额变化可溯源。

### 阶段 6 · 用户端视觉精致化 ⭐
- 按第 9 节把用户端做到 Vercel/Linear 级精致（**非截图样式**）。
- **验收**：用户端通过设计评审，明显区别于开发者后台；管理端高效可用；两端 token 统一。

### 阶段 7 · 稳定性与部署
- 限流、并发控制、熔断、Docker 生产配置、基础测试。
- **验收**：单 Key 超限返回 429；渠道失败自动切换/熔断；重启数据不丢；README 含部署步骤。

## 20. 关键决策（已拍板，开发直接遵循）

| # | 决策 |
|---|------|
| 1 | 计费 = **上游基准成本 × 倍率 = 扣费，从余额扣**；账本/日志同记 base_cost 与 charge 以算毛利。**不用按次积分、不用直接定单价。** |
| 2 | 第一版只支持 **OpenAI 兼容标准 API Key 上游**。 |
| 3 | 余额为内部高精度点数，对外称「余额」，不强制等价人民币。 |
| 4 | 默认**不开放自助注册**，管理员建号（开关可改）。 |
| 5 | 充值只通过**兑换码 + 管理员手动充值**，无任何在线支付。 |
| 6 | 角色只有 **admin + user**，无上下级。 |
| 7 | 第一版完整支持 **chat 模型**；image 模型做价格展示 + 日志结构预留，按次计费同样乘倍率。 |
| 8 | **用户端 shadcn/ui 求精致，管理端 Ant Design 求效率**，共用设计 token。 |
| 9 | 保留**审计日志**；保留 `channel_models` 关联表与模型名映射；保留 `redeem_records` 独立表。 |
| 10 | 优先桌面端，移动端做到可用。 |
