## 四、本轮新发现缺陷与修复步骤（下一轮必做）

> 以下按严重程度排序。每项均给出**精确文件、行号、改法**，请 agent 严格照做，不要自由发挥。

### 🔴 N1（严重）：Anthropic 渠道是「伪流式」，丧失 SSE 实时性

**现象**：OpenAI 渠道是真透传（handler 用 `CopyAndCapture` 边读边写客户端）；但 Anthropic 渠道在 `AnthropicAdapter.OpenChatStream` 内调用 `ConvertAnthropicStream(resp.Body)`，该函数用 `scanner.Scan()` **循环读到上游 EOF 才返回**，再把完整结果 `io.NopCloser(bytes.NewReader(converted))` 包回 `resp.Body`。
**后果**：用 Claude 渠道时，客户端要等上游**全部生成完**才一次性收到全部内容，打字机效果消失，长回答体验等同卡死。

**根因定位**：
- `backend/internal/upstream/anthropic_adapter.go:84-105`（`OpenChatStream`）
- `backend/internal/upstream/anthropic_adapter.go:199-264`（`ConvertAnthropicStream` 阻塞读完）

**修复步骤**：
1. 将 `ConvertAnthropicStream` 从「读完返回 `[]byte`」改为「边读边转的流」。推荐用 `io.Pipe`：
   - 新增函数 `func StreamAnthropicToOpenAI(upstream io.ReadCloser) io.ReadCloser`，内部起 goroutine：`bufio.Scanner` 逐行读上游 → 每遇到 `content_block_delta` 立即 `writeOpenAIStreamChunk` 写入 `pipeWriter` → flush；末帧写 usage + `data: [DONE]`；结束 `pw.Close()`。
2. `OpenChatStream` 改为：
   ```go
   resp.Body = StreamAnthropicToOpenAI(resp.Body) // 不再 ReadAll/NopCloser
   resp.Header.Set("Content-Type", "text/event-stream")
   resp.Header.Del("Content-Length")
   return resp, nil
   ```
3. handler 层 `CopyAndCapture` 已是边读边写，无需改动；但需确认 `CopyAndCapture` 内对 `http.Flusher` 调用 `Flush()`（见 N1-附）。
4. 保留旧的 `ConvertAnthropicStream`（同步版）仅供单测使用，或在 `anthropic_adapter_test.go` 中改测新流式函数。
5. **验证**：对 Claude 渠道发 `stream:true` 请求，用 `curl -N` 观察是否逐帧到达（而非一次性）；同时确认末帧 usage 仍被 `ExtractStreamUsage` 正确计费。

**N1-附**：核对 `service.CopyAndCapture`（在 `gateway_service.go` 或同包）是否每写一段就 `flusher.Flush()`。若没有，OpenAI 真流式同样会被 Go 的 `http.ResponseWriter` 缓冲拖慢。请一并加 `if f, ok := w.(http.Flusher); ok { f.Flush() }`。

### 🟠 N2（中等）：管理端 gemini 是「假选项」，选了必失败

**现象**：管理端供应商下拉有 4 项（`admin-page-utils.tsx:19-24`）：`openai / claude / gemini / custom`。但后端 `ProviderForType`（`provider.go:14-21`）只对 `anthropic/claude` 返回 `AnthropicAdapter`，**其余全部 fallthrough 到 `OpenAIAdapter`**。
**后果**：管理员建一个 `provider_type=gemini` 的渠道后，网关会用 **OpenAI 协议**去打 Gemini 端点，请求格式不匹配，调用必然失败，且错误信息令人困惑。`custom` 同理（语义未定义）。

**修复步骤（二选一，推荐方案 A）**：
- **方案 A（先收窄，稳妥）**：删除前端 `gemini`、`custom` 两个选项，仅留 `openai`、`claude`。改 `admin-page-utils.tsx:19-24`。
- **方案 B（补能力）**：实现 `GeminiAdapter`（`backend/internal/upstream/gemini_adapter.go`），完成 OpenAI↔Gemini `generateContent` 请求/响应/SSE/usage 转换，并在 `ProviderForType` 增加 `case "gemini"`。`custom` 含义需先与产品确认再做。
- 无论哪个方案，都要**保证前端下拉项与后端 `ProviderForType` 的 case 严格一一对应**。

### 🟡 N3（轻）：用户详情页前端硬编码 `1, 100`，超 100 条不可见

**现象**：`frontend/admin/src/pages/users.tsx:49-51` 调 `adminUserAPIKeys(id,1,100)`、`adminUserLogs(id,1,100,...)`、`adminUserLedger(id,1,100,...)`。后端虽已支持真分页，但前端写死第 1 页 100 条。重度用户日志/账本超 100 条时，详情页统计（累计消费/充值）会算少。
**修复步骤**：
1. 给详情页三张表各加独立 `Pager` 状态 + AntD `Table pagination`，分别请求对应页。
2. 「累计消费/累计充值」不要在前端用当前页数据 `reduce`；改为后端在 `/users/{id}` 或新增 `/users/{id}/summary` 直接返回聚合值（避免分页后算错）。

### 🟡 N4（轻）：`max_tokens` 默认值 2048 硬编码两处

**现象**：`anthropic_adapter.go:134` 与 `gateway_service.go:174` 各写死 `maxTokens = 2048`。Anthropic 请求**必填** `max_tokens`，但 2048 对长输出会被无声截断。
**修复步骤**：把默认值提到 config（如 `DEFAULT_MAX_TOKENS`，给一个更合理的值如 4096），两处统一引用；并在文档说明该默认值的影响。

---

### 🟡 N5（轻）：provider 字面值不统一（claude vs anthropic）

**现象**：前端运营值用 `claude`，后端 `ProviderForType` 同时认 `anthropic` 和 `claude`。双字面值长期看是隐患（统计、迁移、排错易混）。
**修复步骤**：选定唯一规范值（建议 `anthropic`），前端下拉 value 改为 `anthropic`，后端保留 `claude` 作为**兼容别名**并加注释说明「历史兼容，新数据一律 anthropic」。DB 内存量数据如有 `claude` 可写一次性迁移脚本。
