- `src/styles.css` 改为 `@import "tailwindcss";` + `@theme {}` 主题块（颜色/圆角/阴影/字体已迁入）。

## 二、三条红线核查结果

### 🟢 红线 1 · 计费模型（倍率 + 余额 + 预占结算）—— 基本达标，有缺陷

**✅ 做对的（已验证）：**
- 公式正确：`charge = base_cost × rate_multiplier`，token 与 per_call 两种算法都实现（`internal/billing/calculator.go`），并有单元测试佐证（`1.3 倍`、`per_call ×1.2` 用例通过）。
- 预占冻结：Redis `INCRBY frozen:<uid>` 原子预占，不足回滚并返回 **402**（`billing/reserve.go`）。**并发测试已验证：50 并发、余额仅够 10 个时恰好 10 个成功**（`reserve_test.go`）。
- 真实结算：在**单个 Postgres 事务**内完成 `SELECT...FOR UPDATE 行锁 → 写 gateway_requests → 扣余额（带 balance>=charge 守卫）→ 写 balance_ledger`（`repository/gateway.go:144-193`）。
- 并发防扣负：Redis 预占 + DB `WHERE balance>=$2` 双层防护。
- 失败释放预占（`defer Release`）、流式结束补结算、usage 缺失标记 `is_estimated` —— 均已实现。
- 毛利可算：`gateway_requests` 与 `balance_ledger` 均持久化 `base_cost / rate_multiplier / charge`，毛利 = Σcharge − Σbase_cost 可计算。

**⚠️ 缺陷（按严重度）：**
1. **【高·收入泄漏】per_call 计费分支未接入网关。** `CalculatePerCallCharge` 函数存在但**生产路径从不调用**，网关层（`gateway_service.go`）对所有模型硬编码走 token 计费。结果：`billing_mode='per_call'` 的文生图模型会被当 token 计费 → 计费为 0 或错误。
2. **【高·计费偏差】"tokenizer" 是 `len(text)/4` 字节估算**（`billing/tokenizer.go`），非真实 BPE。对中文严重低估（约真实值 30-50%）；流式补结算时还把 **SSE 协议帧 + JSON 整体喂给估算器**，未剥离结构，误差方向不可控。usage 缺失的流式请求将系统性误差。
3. **【中·账本完整性】** 余额不足时 `UPDATE...WHERE balance>=charge` 返回 0 行 → 整事务回滚，连 `gateway_requests` 也丢失 → 出现"用户已得到上游响应，但平台无任何账单留痕"。
4. **【中·Redis 漏冻】** `frozen:<uid>` key 无 TTL，进程崩溃会永久冻结用户余额，无回收兜底。
5. **【低·价格快照不全】** 仅快照 `rate_multiplier`，未快照 `input/output_price_per_1k`。价目表改动后无法重放历史每-token 单价，对账严谨度不足。

### 🔴 红线 2 · 用户端精致 UI —— **不合格（1.5/10）**

这是本次验收**最严重的问题**。用户端（`frontend/user`）实际是一个 **约 170 行的单文件 PoC**，远未达到"Vercel/Linear 级精致"要求：

- **shadcn/ui：0 个组件。** `src/components/ui/` 空目录，无 `components.json`，无 `cn()` 工具。
- **TailwindCSS：0 处使用。** 无 `tailwind.config`、无 `postcss.config`、无 `@tailwind` 指令。依赖（tailwindcss/cva/tailwind-merge）**装了但完全没用** —— 典型"骨架占位提交"。
- **路由：0 条。** `src/routes/` 空，未装 `react-router-dom`，所有内容堆在一个 `App()` 里，刷新即丢登录态。
- **登录方式破防：让终端用户手动粘贴 JWT 到输入框**（`main.tsx:57-63）—— 这恰恰是"开发者后台"形态，直接违反红线。
- **缺失页面：** login / dashboard / api-keys / usage / models / redeem / announcements / settings **全部没有独立实现**。
- **无 API Key 管理 UI**、**无 usage Tab（每日/按模型/扣费/日志）**、**无趋势图（未装 recharts）**、**StatCard 是灰盒子**、**EmptyState 是纯文字**、**无深色主题、无动效、无骨架屏**。

> **判定：用户端需推倒重写。** 唯一达标项是"用了顶部导航而非左侧裸按钮竖排"。

### 🟢 红线 3 · 功能范围（精简）—— 完美守住

- ✅ 无任何在线支付网关（alipay/wechat/stripe/easypay 全工程 0 命中）。
- ✅ 无多级代理/分销/返佣/团队/租户/子账号（相关关键词 0 命中，users 表无 parent_id/tenant_id）。
- ✅ 角色仅 admin + user（迁移 CHECK 约束 + 中间件双重保证）。
- ✅ 未引入任何第三方支付 SDK。
- ⚠️ 唯一瑕疵：充值的两条通道中，"兑换码"已实现，但"**管理员手动充值**"接口缺失（见下）。

## 六、给运营者（你）的行动建议

1. **后端可以保留并继续**，不需要重做 —— 地基和计费内核质量达标。
